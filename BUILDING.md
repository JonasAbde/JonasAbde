# Building Method

This is how I approach projects, especially AI and automation workflows.

## 1. Start with the workflow

I first try to understand the actual workflow:

- What information comes in?
- Who needs to act on it?
- What decision must be made?
- What happens manually today?
- What is repetitive or easy to miss?

## 2. Define the useful outcome

Before building, I try to define what the tool should actually improve.

Examples:

- Save time
- Reduce missed follow-ups
- Make decisions clearer
- Improve overview
- Turn messy information into structured actions

## 3. Prototype small

I prefer small prototypes that prove one useful idea.

A first version does not need to be perfect. It needs to show whether the workflow can become more useful.

## 4. Keep humans in control

For business, customer, finance, or sensitive workflows, AI should support the human instead of silently making important decisions.

I prefer AI systems that:

- Prepare drafts
- Organize information
- Suggest next steps
- Highlight risks
- Make work easier to review

## 5. Test against real examples

A workflow is only useful if it works with real or realistic examples.

I test by asking:

- Does this handle messy input?
- Does it produce a useful output?
- Is the result understandable?
- Can a person review it quickly?
- What can go wrong?

## 6. Document the learning

Every project should explain:

- Problem
- Solution idea
- Tools
- What was built
- What worked
- What failed
- What should be improved next

## 7. Improve iteratively

The best systems usually come from iteration:

Prototype → test → improve → document → repeat.
